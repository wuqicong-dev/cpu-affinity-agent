# CPU绑核相关抖动问题案例集

# 案例1：Decode节点抖动（Host与GC优化）

## 一、问题现象

客户拓展过程中，Decode节点出现抖动，影响交付节点正常运行；每几千个step中会出现几十次100ms以上的抖动，正常单step耗时约67ms。

## 二、问题分析

1. 通过复现客户场景、隔离Host/Device抖动、GC分析等步骤，定位抖动核心来源。

2. 添加打点代码监测，发现Host抖动占总抖动时间的39%，需优先进行Host侧优化。

3. 抖动类型区分：不规律大抖动怀疑是OS调度异常导致，规律性抖动怀疑是GC（垃圾回收）引发。

4. GC日志分析：每隔60+个step会出现一次0.4ms的Young GC，确认GC是规律性抖动的核心原因，需同步优化GC策略。

## 三、解决方案

1. 线程干扰优化：采用自动绑核、关闭numa平衡、线程隔离等手段，减少Host侧线程争抢，降低OS调度抖动。

2. GC优化：在Decode执行前关闭GC，待每次Decode的Device侧运行时，手动触发Young GC，利用Device运行时间掩盖GC耗时，消除规律性抖动。

## 四、优化效果

经上述优化，Decode节点抖动问题得到有效解决，恢复正常运行耗时。

# 案例2：绑核引发的气泡及profiling数据失真问题

## 一、问题背景

在A3 A+K（鲲鹏920B）及客户实际场景中，开启绑核策略后，出现气泡问题及性能profiling数据失真，影响性能分析准确性。

## 二、问题现象

通过torch_npu.profiler插桩分析Decode阶段单次forward的Host侧性能时，采集到的profiling数据中存在明显气泡，导致性能分析偏差。

## 三、问题分析

1. 初步怀疑GC导致气泡，关闭GC后，气泡问题有所缓解，但未完全消除，排除GC为核心根因。

2. 进一步分析业务线程、OMP线程、msprof线程的干扰情况，定位核心问题：开启profiling采集时，EngineCore下发线程会创建多个msprof线程。

3. 当前绑核策略仅为EngineCore分配1个CPU核，导致fork出的msprof线程与EngineCore线程争抢CPU资源，引发气泡，进而导致profiling数据失真。

## 四、解决方案

开启profiling采集时，为EngineCore下发进程分配多个CPU核资源，避免msprof线程与EngineCore线程争抢资源，消除气泡问题，确保profiling数据的准确性。

# 案例3：Vllm PD分离架构下的Host抖动问题（6类场景）

客户使用Vllm PD分离架构时，出现6类Host抖动问题，以下重点梳理与绑核相关的2类核心场景，其余场景简要说明。

## 核心场景1：算子下发延迟抖动

1. 现象：偶现算子下发后，延迟空等2ms才执行。

2. 分析：观察程序运行状态，发现进程被绑定到错误的CPU核，属于prof开启时的绑核bug。

3. 解决方案：prof开启后，手动执行绑核操作，修复绑核bug，避免进程绑定错误。

## 核心场景2：sync延迟抖动

1. 现象：算子已执行完成，但sync延迟结束，存在耗时冗余。

2. 分析：代码打点监测发现，rts下发缓慢，存在800+us抖动，且出现大量pagefault（缺页中断）；根源是分配的数组（4M大小）过大，引发缺页中断。

3. 解决方案：启动RTS时，为代码段、数据段配置2M大页内存，减少缺页中断，解决sync延迟问题。

## 其他场景简要说明

- GMM算子shape负载不均衡：调整负载均衡策略解决。

- npu_moe_finalize_routing和npu_quant_matmul算子慢：优化算子执行逻辑。

- D2H内存拷贝时间过长：优化内存传输策略。

- GMM算子冗余infershape：去除冗余infershape操作，减少耗时。

# 案例4：商用模型加压后网卡流量放大及跨P抖动问题

## 一、问题现象

客户商用模型上线加压（增加请求数和序列长度）后，新出现抖动问题，伴随网卡流量放大现象：

- iperf测试：客户端在P0节点，裸数据性能10GB/s；

- P0节点DDR写带宽达15GB/s（含网络协议栈开销）；

- hccs上运行流量20GB/s，hccs效率约75%，数据流量放大1.5倍。

## 二、问题根因

跨片数据流量中，driver进程从P0到P1的copy_to_user操作占比达78%，是导致流量放大、抖动的核心原因，且CPU负载过高。

## 三、解决方案

### 1. 多流多队列+DMA直通传输优化

使能non-cacheable DMA直通传输，将KV Cache传输至4个CPU的本地内存，提升跨P流量传输效率，降低memcpy开销，减少流量放大。

### 2. 负载均衡优化

将网卡报文接收队列分组后，绑定到4个CPU；Decode节点在卸载Prefill任务时，预先分配KV Cache至目的Driver/Worker组所在CPU，实现负载均衡。

### 3. CPU绑核策略

采用4X4对称绑核方式：网卡接收队列中断绑定到CPU Socket，Driver、Worker、Forward进程均绑定到相同P节点，进一步降低线程干扰和跨P传输耗时。

## 四、跨P数据搬运优化

通过Non-cacheable DMA实现跨P数据搬运：多流多队列方案将队列中断绑定到指定CPU，CPU在本地内存分配DMA buffer，使能网卡DMA直通传输KV Cache到本地内存，提升HCCS/QPI传输效率，降低CPU负载，解决抖动问题。
